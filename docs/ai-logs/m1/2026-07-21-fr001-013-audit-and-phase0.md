## Key findings (see full per-FR table delivered to the owner in-chat)

- PROGRESS.md's claim that docs/srs and docs/design are "3-line TODO stubs" is
  **stale/incorrect** — all FR-001..FR-013 docs and 01-07 design docs are
  substantive v2 specs. The shipped backend/frontend implement an earlier,
  materially simpler design than these v2 docs describe (spans FR-002, FR-004,
  FR-005, FR-007, FR-009, FR-010, FR-011, FR-012). This is a spec-vs-code
  conflict needing an explicit owner decision, not a coding fix.
- No FR is fully REAL_E2E end-to-end except FR-003 backend+frontend Garage
  CRUD (with one exception noted below). Booking wizard (FR-004) is 100% mock
  data end-to-end on the frontend. Payment (FR-005) has zero backend
  implementation (no PaymentController, no VNPAY/payOS code, only a static
  VietQR image-URL string). Admin FR-010/011/012/013 frontend is completely
  absent (no admin routes/pages exist at all in the current Front-end/src).
- **CRITICAL, newly discovered, currently live**: `CustomerController`
  (`Back-end/src/main/java/com/autowashpro/controller/CustomerController.java`)
  and `CustomerServiceImpl` are not covered by any role-scoped matcher in
  `SecurityConfig.java` (lines 35-47), so they fall through to
  `.anyRequest().authenticated()`. Any authenticated CUSTOMER can list every
  customer's PII (`GET /api/v1/customers`), read any single account by id
  (IDOR), and — most severely — `PUT /api/v1/customers/{id}` accepts a raw
  client-supplied `passwordHash` field and writes it verbatim with no
  re-hash and no ownership check (`CustomerServiceImpl.java:61`). Since
  ADMIN/STAFF share the same `customers` table and the seeded ADMIN phone is a
  hardcoded constant (`+84900000002`), a self-registered customer can look up
  the admin's id, overwrite its password hash, and log in as ADMIN — a full
  auth-bypass / privilege-escalation path. `DELETE /api/v1/customers/{id}` has
  the same missing-ownership-check problem. This is independent of, and more
  severe than, the already-tracked SystemAccountSeeder hardcoded-password
  item. Not yet fixed — flagged to the owner for an explicit go/no-go before
  patching, per the phase-gate process the owner specified.
- Also reconfirmed still-open items already in PROGRESS.md (SystemAccountSeeder
  hardcoded legacy credentials, now redacted; no Back-end/src/test tree; payment gap) and
  found new ones: FR-003's "unauthorized vehicle update -> 403" acceptance
  criterion is not met (VehicleServiceImpl throws plain RuntimeException,
  caught by the generic handler, so it returns 500 instead of 403); FR-008
  voucher redemption trusts a client-supplied `pointsCost` instead of
  server-enforcing the fixed per-voucher-type point price (a customer can
  redeem a high-value voucher for `pointsCost=1`); FR-009's STAFF-cannot-
  self-complete rule (BR-024) does not exist — STAFF/ADMIN is currently the
  *only* role that can drive a booking to COMPLETED, with no payment
  verification, enabling internal fraud; no `@Version`/locking on
  `Customer.accumulatedPoints` or `Voucher.status`, so concurrent
  redeem/apply requests can double-spend.

## Implementation performed after the audit (same work item)

Two fixes with all prerequisites already available were made immediately,
per the owner's own instruction to implement Phase 0 work that doesn't need
new decisions, while everything else (Phase 1+) waits for explicit approval:

1. **CRITICAL security fix** — `Back-end/src/main/java/com/autowashpro/config/SecurityConfig.java`
   now requires `hasRole("ADMIN")` for `/api/v1/customers/**` (previously
   unmatched, falling through to `.anyRequest().authenticated()`).
   `CustomerServiceImpl.updateCustomer()` no longer honors a client-supplied
   `passwordHash`; `CustomerMapper.toEntity()` explicitly ignores it too; the
   field was removed from `CustomerRequest` entirely since both write paths
   ignored it. Live-verified (not just read) after restarting the backend:
   a forged CUSTOMER-role JWT now gets `403` on `GET /api/v1/customers` and
   `PUT /api/v1/customers/1` (previously `200`, i.e. the full IDOR/account-
   takeover path); a forged ADMIN-role JWT still gets `200` on the same GET,
   proving the fix scopes access rather than breaking the endpoint; unrelated
   endpoints (`GET /api/v1/vehicles`, `GET /api/v1/catalog/branches`) were
   re-checked with a real CUSTOMER token and are unaffected.
2. **Phase 0 CTA identity-choice flow** — new
   `Front-end/src/features/landing/BookingEntryContext.tsx` +
   `Front-end/src/features/landing/components/BookingEntryModal.tsx` provide
   a `useBookingEntry()` hook and a modal with three choices (Sign in /
   Register / Continue as guest). `Navbar.tsx`'s header CTA and
   `BookingWidget.tsx`'s in-hero CTA (the only two places that previously
   called `navigate('/booking')` directly — verified by grep, every other
   landing CTA just scrolls to the in-page widget) now open this modal
   instead. `LoginPage.tsx` now reads an optional `mode` from router state so
   "Register" opens straight to the register tab; both paths reuse the
   page's existing `from.pathname` redirect-after-auth mechanism to return to
   `/booking`. "Continue as guest" preserves the previous direct-to-wizard
   behavior and any prefill state. Added `entryModal.*` i18n keys to both
   `vi` and `en` `landing.json` (LoginPage itself predates i18n and remains
   hardcoded Vietnamese, consistent with its existing pattern — not touched).
   Live-verified in Chrome: all three choices produce the correct navigation
   (Sign in -> `/login` login tab, Register -> `/login` register tab with
   `Đăng ký` pre-selected, Guest -> `/booking` step 1/6 unchanged).
   `npx tsc --noEmit` and `npm run build` both clean after the change.

Explicitly NOT done in this work item (Phase 1+ scope, needs owner decisions
first): rewiring the booking wizard off mock data, implementing FR-005
payment (VNPAY vs payOS is still undecided), reconciling the FR-002
Firebase-token-vs-phone+password spec conflict, the FR-008 client-supplied
`pointsCost` value-manipulation bug, the FR-009 STAFF-can-self-complete
authorization gap, or adding a `Back-end/src/test` tree.
