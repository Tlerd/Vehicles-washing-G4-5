## A. FR-003 authorization fix (implemented, tested, verified)

**Root cause** (systematic-debugging Phase 1‚Äì2, direct code evidence, no
speculation): `VehicleServiceImpl.findOwnedVehicle()` ‚Äî the single helper
backing `updateVehicle`, `deleteVehicle`, and `setDefaultVehicle` ‚Äî threw a
plain `new RuntimeException("Unauthorized vehicle access or vehicle not
found.")` when a vehicle didn't exist or wasn't owned by the caller.
`GlobalExceptionHandler` only had specific `@ExceptionHandler` mappings for
`ResourceNotFoundException`(404)/`BadRequestException`(400)/
`ConflictException`(409)/`UnauthorizedException`(401); a plain
`RuntimeException` fell through to the catch-all `Exception.class` handler,
returning 500 instead of the FR-003 acceptance criterion's required 403.

**Fix (TDD, red‚Üígreen, verified with `mvn -f Back-end/pom.xml test`):**
- Added `Back-end/src/main/java/com/autowashpro/exception/custom/ForbiddenException.java`
  (mirrors the four existing custom exceptions).
- Added `handleForbidden` to `GlobalExceptionHandler.java`, mapping to
  `HttpStatus.FORBIDDEN`.
- Changed `findOwnedVehicle()` in `VehicleServiceImpl.java` to throw
  `ForbiddenException` instead of `RuntimeException`.
- Added `Back-end/src/test/java/com/autowashpro/service/impl/VehicleServiceImplTest.java`
  (3 tests: `updateVehicle`/`deleteVehicle`/`setDefaultVehicle` throw
  `ForbiddenException` when the vehicle isn't owned by the caller).
- Added `Back-end/src/test/java/com/autowashpro/exception/handler/GlobalExceptionHandlerTest.java`
  (1 test: `handleForbidden` returns HTTP 403).
- Watched RED first: a compile-error RED (handler method missing), then a
  correct runtime RED (`RuntimeException` thrown, not `ForbiddenException`) ‚Äî
  both confirmed before implementing the fix.

**Evidence:** `mvn -f Back-end/pom.xml test` ‚Üí `BUILD SUCCESS`, **11 tests
run, 0 failures, 0 errors, 0 skipped** (7 pre-existing `AuthServiceImplTest` +
3 new `VehicleServiceImplTest` + 1 new `GlobalExceptionHandlerTest`).

**Independent review** (background Workflow, 2 agents, both `approved: true`):
- `code-reviewer`: fix is correct and covers all three call sites; flagged
  (informational/MEDIUM) that no MockMvc/integration test proves the fix at
  the actual HTTP layer (repo has zero MockMvc/`@SpringBootTest` tests
  anywhere ‚Äî consistent with the existing suite, not a regression) and that
  no `docs/ai-logs`/`PROGRESS.md` entry existed yet at review time (this log
  and the `PROGRESS.md` update close that gap).
- `security-reviewer`: confirmed collapsing "not found" and "not owned" into

  below): two ownership checks of the identical bug class in
  `BookingManagementService` (400 instead of 403), and the rest of
  `VehicleServiceImpl`'s validation paths still throwing bare
  `RuntimeException` ‚Üí 500.

## B. FR-001/FR-002 documentation correction (implemented)

Read `AuthServiceImpl`, `RegisterRequest`/`LoginRequest`, `LoginPage.tsx`,
and `AuthContext.tsx` directly. Corrected both SRS docs, which previously
described an aspirational "Firebase-identity-token" design:

- **FR-002 (login)**: login is phone+password only against the backend's own
  bcrypt store ‚Äî **zero Firebase involvement**. The same
  `POST /api/v1/auth/login` endpoint issues STAFF/ADMIN JWTs too (they're
  rows in the same `customers` table), contradicting the old doc's "does not
  issue staff or admin credentials" claim.
- **FR-001 (registration)**: phone+password is always collected regardless of
  provider; Firebase verifies phone (OTP path) or email (Google path) once,
  at registration only; `POST /api/v1/auth/register` returns no JWT ‚Äî the
  frontend calls `/login` immediately after.
- Found and cited `docs/superpowers/specs/2026-07-21-google-signin-registration-design.md`
  (an owner-approved design from earlier the same day) which shows two things
  I had initially mis-labeled as "gaps" ‚Äî no deferred phone for Google users,
  no cross-provider account linking ‚Äî are actually **deliberate, approved
  product decisions**, not unfinished work. Corrected the doc framing
  accordingly before finalizing, citing that design doc.
- Fixed dangling `../refactor/...` relative links in both docs' Source Rules
  sections (no `docs/refactor/` directory exists; real paths are
  `docs/srs/06-BUSINESS-RULES-V2.md` and `docs/plans/PLAN-V2-LAM-LAI-FE.md`).

**Independent review** (`security-reviewer`, `approved: true`): verified the
rewritten docs' claims against the actual code; found no inaccuracies in the
rewritten text itself. Surfaced two **pre-existing, unrelated** findings in
`AuthServiceImpl` (phone-existence check runs before Firebase-token
verification, creating a minor registration-phone enumeration oracle; a
possible raw DB constraint violation on a second no-email registration due to
SQL Server's single-NULL unique-index semantics) ‚Äî both recorded in the
FR-004 plan doc's "related findings" section, not fixed (unrelated to the
requested doc correction).

## C. FR-004 audit + plan (audit complete; no implementation this pass)

Dispatched a 6-agent parallel background Workflow per AGENTS.md's mandatory
role-dispatch policy: `backend-lead`, `frontend-lead`, `testing-lead`,
`security-reviewer` (FR-004 audit) run concurrently with `code-reviewer` +
`security-reviewer` (independent review of the Part A/B changes, above).

**Key audit findings** (full detail in
`docs/superpowers/specs/2026-07-21-fr004-real-booking-implementation-plan.md`):
- The booking wizard frontend is **100% mock** across all 6 steps; no step
  makes a real HTTP call.
- The backend already has a working, authenticated booking API, but it
  implements a materially simpler, earlier design than the FR-004/FR-005 v2
  spec (no 15-minute slot/bay model, no soft-holds, no VNPAY, `PENDING` not
  `PENDING_DEPOSIT`). The spec's own documented API paths and a cited
  `docs/refactor/` directory don't match this repository's reality.
- Guest booking remains architecturally blocked by three independent, stacked
  barriers (authenticated-only route matcher, server-side JWT-derived
  `customerId` override, `NOT NULL` FK) ‚Äî this **reconfirms** the
  2026-07-21 guest-booking SQL-migration rejection; nothing has changed.
- A live, unreconciled `VehicleSize` taxonomy mismatch exists between the
  backend's persisted 4-tier enum and the frontend wizard's 3-tier model.
- Test tooling for what FR-004 really needs (Testcontainers,
  `spring-security-test`, Vitest, Playwright) is entirely absent today; none
  should be added silently.
- Found (not fixed) two authorization bugs of the identical class just fixed
  in Part A, in `BookingManagementService` ‚Äî see Part A above and the plan
  doc.

**Why no FR-004 code was written**: the user's own instruction conditioned
implementation on "when the contract is clear and secure." The audit itself
surfaced five genuine, owner-level decisions (target contract version for
Phase 1, guest-booking route handling, `VehicleSize` taxonomy, combo-service
scope, test-tooling additions) that are not mine to decide unilaterally and
that change what "safe, well-defined" implementation would even mean. The
plan document enumerates all five explicitly and proposes a recommendation
for each, plus a 3-phase incremental path once they're answered.

**A process note from the audit agents themselves**: each of the 4 audit
agents reported that their own sub-runtime exposed only Read/Glob/Grep/
StructuredOutput ‚Äî no further Task/Agent dispatch tool ‚Äî so they could not
literally spawn their own named specialist children (e.g. `backend-lead`
could not dispatch `backend-api-specialist`/`backend-data-specialist`/
`backend-security-specialist` as separate agents) and instead performed the
equivalent analysis directly, organized under those three lenses. This is
recorded as an observed runtime limitation, not a silently-skipped policy
step, consistent with this repository's existing pattern of recording such
discovered limitations (e.g., the Codex sandbox-helper blocker already in
`PROGRESS.md`).
